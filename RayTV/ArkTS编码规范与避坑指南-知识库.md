# ArkTS编码规范与避坑指南 - 知识库

## 核心原则

### 1. 类型安全优先
- 禁止使用`any`和`unknown`类型
- 明确变量、函数参数和返回值的类型
- 充分利用静态类型检查避免运行时错误
- 使用类型保护机制进行运行时类型检查

### 2. 性能优化导向
- 基于AOT编译特性编写高性能代码
- 避免在循环中频繁创建对象
- 合理使用内存管理，避免内存泄漏
- 优化渲染性能，避免不必要的渲染

### 3. 状态驱动UI
- 遵循"状态驱动UI更新"的现代前端开发范式
- 合理使用@State、@Prop等装饰器管理状态
- 避免在build方法中执行复杂计算
- 使用@ObservedV2 + @Trace优化复杂对象状态管理

### 4. 组件化开发
- 采用组件化架构，提高代码复用性和可维护性
- 每个组件只负责一个明确的功能
- 合理使用@Entry和@Component装饰器
- 提取可复用组件，减少重复代码

## 编码规范

### 1. 代码格式
- 使用4个空格进行缩进，不使用制表符
- 每行代码建议不超过120个字符
- 在逻辑块之间使用空行分隔，提高可读性
- 使用K&R风格，左大括号与语句同一行，右大括号单独一行

### 2. 命名规范
- 变量名：使用小驼峰命名法（camelCase）
- 常量名：使用全大写加下划线（UPPER_SNAKE_CASE）
- 函数名：使用小驼峰命名法，动词开头
- 类名/接口名：使用大驼峰命名法（PascalCase）
- 文件命名：使用小驼峰命名法，与导出的主要类或函数名保持一致
- 布尔变量：使用is、has、can等前缀

### 3. 语法使用
- 数值类型：统一使用number，大整数需显式标注BigInt
- 字符串：优先使用模板字符串（`）而非字符串拼接
- 布尔值：避免使用0、1等数字代替布尔值
- null与undefined：明确区分使用场景，null表示有意的空值，undefined表示未定义
- 变量声明：优先使用const，其次使用let，禁止使用var

### 4. 函数与方法
- 明确参数类型和返回值类型
- 函数长度不应超过50行
- 参数数量不应超过5个，超过时使用参数对象
- 优先使用箭头函数，尤其是在回调函数中
- 避免函数产生不可预测的副作用

### 5. 类与对象
- 明确使用public、private、protected访问修饰符
- 使用参数属性简化构造函数
- 遵循单一职责原则，每个类只负责一个功能领域
- 优先使用组合而非继承
- 避免类过于庞大，超过500行

### 6. 接口与类型
- 使用interface定义对象结构
- 使用type定义联合类型或复杂类型
- 接口应该尽可能小且专注
- 优先使用字面量联合类型替代枚举类型
- 为联合类型提供明确的类型守卫函数

## 常见错误与避坑

### 1. 语法错误
- JSDoc注释格式错误：缺少@符号
- 闭合括号位置错误：接口或类的闭合括号没有单独成行
- 属性换行错误：多个属性挤在同一行
- 注释全角字符：注释中使用了全角问号（？）而不是半角问号（?）

### 2. 类型系统错误
- 类型作为值使用：将类型（如boolean, string, number）作为值使用
- 缺少必要属性：对象缺少接口中定义的必要属性
- 多余属性：对象包含接口中未定义的属性
- 类型不兼容：方法参数类型与定义不匹配

### 3. 工具类使用错误
- Logger调用错误：Logger方法调用缺少必要的tag参数
- fs模块不存在：尝试使用Node.js的fs模块
- context对象未定义：尝试使用context对象，但未正确获取或定义

### 4. 框架API错误
- router类型不兼容：使用了不存在的router.PushOptions和router.ReplaceOptions类型
- Preferences API不兼容：尝试使用不存在的Preferences.keys属性
- 已弃用API：使用了已弃用的@ohos.arkui等API

### 5. 性能陷阱
- 频繁DOM操作：避免频繁操作DOM，使用虚拟DOM
- 不必要的计算：避免在渲染过程中进行复杂计算
- 内存泄漏：避免循环引用和未释放的事件监听器
- 频繁状态更新：避免频繁触发状态更新

## 最佳实践

### 1. 代码组织
- 分层架构：components/（组件层）、services/（服务层）、models/（模型层）、pages/（页面层）
- 功能模块化：按业务功能划分模块目录
- 单一职责：每个文件只负责一个明确的职责
- 避免循环依赖：通过依赖注入、接口隔离等方式避免模块间的相互引用

### 2. 错误处理
- 使用try-catch处理可能抛出异常的代码
- 定义明确的错误类型和错误码
- 合理使用throw和rethrow传播错误
- 提供错误恢复机制和重试逻辑
- 正确处理Error对象，避免直接使用字符串作为错误

### 3. 异步编程
- 使用try-catch包装async/await处理异步错误
- 避免Promise的深度嵌套
- 合理使用Promise.all、Promise.race等方法
- 避免在UI线程中使用await执行耗时操作

### 4. 装饰器使用
- @State：组件内部状态管理，状态变化触发UI更新
- @Prop：父组件向子组件传递数据，单向数据流
- @Link：父子组件双向数据绑定
- @ObservedV2 + @Trace：响应式状态管理组合，用于复杂对象状态管理
- @Entry：用于标记应用入口组件
- @Component：用于定义所有可复用的UI组件

### 5. 测试与调试
- 为关键功能编写单元测试
- 使用Jest、Mocha等标准测试框架
- 使用快照测试验证组件输出
- 使用性能分析工具找出性能瓶颈
- 定期进行代码审查，检查代码是否符合ArkTS规范

## 性能优化

### 1. 内存管理
- 使用ResourceManager类管理资源生命周期
- 实现dispose方法显式释放资源
- 对共享资源使用引用计数机制
- 使用弱引用管理临时对象
- 对频繁创建销毁的对象使用对象池

### 2. 渲染优化
- 使用ForEach渲染列表，提供唯一键值
- 使用cachedCount缓存列表项提升性能
- 明确设置列表方向（Axis.Vertical或Axis.Horizontal）
- 避免在build方法中创建新对象
- 避免在渲染方法中使用匿名函数

### 3. 状态更新优化
- 使用深度比较替代JSON.stringify判断数据变化
- 实现shouldUpdate方法避免不必要的状态更新
- 对多个状态更新使用批量更新机制
- 对频繁的状态更新使用防抖或节流
- 使用异步更新避免阻塞UI线程

## API兼容性

### 1. API版本检查
- 使用@ohos.system获取API版本信息
- 检查当前API版本是否支持所需特性
- 妥善处理版本检查过程中的异常
- 为不支持的特性提供降级方案

### 2. 已弃用API迁移
- 使用@kit.ArkUI替代@ohos.arkui
- 使用字符串值替代枚举值
- 使用@ohos.ability.kits.context替代旧路径
- 逐步替换已弃用API，避免强制升级

### 3. 导入路径统一
- 统一使用@kit.ArkUI导入组件
- 明确导入所需的具体组件
- 对类型定义使用type关键字
- 只导入实际使用的API
- 避免使用通配符导入所有组件
